# ADR-001 — Runtime-Integration, Adapter-Sitz & Reproduzierbarkeits-Seed

**Status:** akzeptiert · **Phase:** 1 (Runtime-Adapter + Session-Modell)

## Kontext
Arkwen soll beliebige Worker (Claude Code, OpenHands, Docker Agent, NemoClaw) hinter einem einheitlichen Session-Modell orchestrieren („OCI für Agenten"). Der Runtime-Adapter-Contract ist das Herzstück: Er entscheidet, ob Arkwen eine echte Runtime-Schicht oder nur ein weiterer Wrapper wird. Grundprinzip: Arkwen abstrahiert **nicht** die innere Loop eines Workers, sondern nur den Rand — Inputs (Mission, Materials, Toolkits), den beobachtbaren Event-Stream und Outputs (Artifacts + Terminal-State).

## Entscheidung 1 — Integrationsmodell: **tiered**
Ein minimaler Pflicht-Contract, den jeder Worker erfüllt, plus optionale Capabilities, die ein Adapter deklariert. Arkwen degradiert graceful.

- **MUSS (jeder Worker):** `lifecycle` (create · start · signal(cancel) · state · reap), `artifacts` (Workbench-Diff + Terminal-State + reason).
- **KANN (deklariert, sonst degraded):** `events.stream` (semantischer Event-Stream statt nur stdout), `control.pause_resume`, `gate.interactive`, `tools.structured`.
- **Kooperativ via Shim/Hooks**, wo die Runtime es sauber hergibt (z. B. Claude Code: PreToolUse-Hooks, Permission-Modi); **observational** (Dateisystem + stdout/stderr + Artefakte) als Boden für „dumme"/fremde Images.

**Warum:** Kompatibel mit guten Integrationen, robust gegenüber fremden Images — keine Abhängigkeit vom idealen Adapter.

## Entscheidung 2 — Adapter-Sitz: **Cell-Shim** (containerd-shim-Muster)
Der Adapter lebt **nicht** im Controller, sondern als schlanker Prozess *in* der Workcell neben dem Worker. Er startet den Worker, tailt dessen native Events, rechnet den Artifact-Diff, brokert Gates und spricht nach außen **nur** das Arkwen-Protokoll über Socket.

- Controller bleibt **runtime-agnostisch** — kennt nur das einheitliche Session-Modell, nie Claude-Code-/OpenHands-Spezialfälle.
- Adapter-spezifische Übersetzung sitzt direkt am Worker (kann sprach-gematcht sein).
- Injektion = statisches Shim-Binary + Socket in das Worker Image mounten (build- oder runtime-injiziert) — analog k8s/containerd.

## Entscheidung 3 — Reproduzierbarkeit: **ab Tag 1 hashbasiert mitschreiben**
„Billig jetzt, unmöglich später." Jeder Production Run schreibt in Phase 1 bereits mit:

- `mission_hash`
- `image_digest`
- `toolkit_versions`
- `workbench_base_commit`
- `workbench_snapshot_ref` **oder** `workbench_mount_ref` *(je nach `materialization_mode`; via ADR-003/004)*
- `materialization_mode` *(snapshot | mount | live-bind; via ADR-004)*
- `adapter_version`
- `policy_version`
- `session_seed`

**Warum:** Ohne diese Seeds bleibt Reproduzierbarkeit eine Behauptung. `session_seed` hält die Nicht-Determinismus-Achse fest (LLM-Runs werden nie bit-genau reproduzierbar, aber annäherbar, vergleichbar, auditierbar).

## Konsequenzen / vorgezogen aus späteren Phasen
- Das **Gate-Primitiv** (pause/resume/inject-decision) ist Phase-1-Contract; Gate-Policies bleiben Phase 2. Präzisierung via ADR-008: `signal(cancel)` ist der universelle MUSS-Contract, `signal(pause|resume)` setzt die KANN-Capability `control.pause_resume` voraus (sonst Workcell-Boundary-Suspension/Freeze).
- Ein **Minimal-Secrets-Pfad** (Env-Injection, wird nie geloggt) ist Phase 1; das volle Secrets-Modell bleibt Phase 3. Redaction-Mechanik siehe ADR-004.

## Nachträgliche Amendments (via ADR-006/007/008/009)
- **Seed erweitert (ADR-006):** +`isolation_contract_ref`, `egress_policy_hash`, `resource_limits_hash`, `image_signature_ref`, `image_provenance_ref`, `secret_scope_set_ref` (Audit-Scope, nie Werte); schema_version-Bump.
- **Seed erweitert (ADR-007):** +`blueprint_digest` (der Pin-Punkt). Bislang „tote" Seeds werden auflösbar: `mission_hash` (Mission-Body content-addressed + secret-frei persistiert), `adapter_version` (Cell-Shim-Binary als Warehouse-Input).
- **Transport (ADR-006):** „Socket" generalisiert zu profil-abhängigem Kanal — unix-domain-socket für runc/gVisor, `virtio-vsock` für Firecracker microVM. Arkwen-Protokoll unverändert.
- **Auflösungs-/Freeze-Punkt (ADR-009 R1):** Alias/Range→Digest wird bei `run.created` genau einmal aufgelöst + geseedet; `provisioning` re-verifiziert nur. `session_seed` kann vom **Outer-Loop-Consumer** in der RunSpec vorgegeben werden (die Runtime nennt keinen Consumer).
