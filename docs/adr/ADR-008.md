# ADR-008 — Outer-Loop Contract (Control Plane)

**Status:** akzeptiert · **Phase:** 5 · **baut auf** ADR-001..005 + Overview · **reconciled durch** ADR-009

## Kontext
Arkwen ist **standalone vollständig**. Diese Phase definiert die **consumer-agnostische Outer-Loop-Contract-Fläche**, über die *ein beliebiger* äußerer Orchestrator (Outer Loop) Arkwen steuert und beobachtet — ohne dass die Runtime einen benannten Consumer kennt oder von ihm abhängt. **Kein Outer Loop ist privilegiert.** (Konkrete Besetzung in unserer Produktfamilie: ELIO = inter-session Outer Loop, Vela = intra-session Inner Loop — siehe `docs/ELIO-reference-consumer.md`. Der Contract selbst nennt keinen Consumer.)

## E1 — Strikt unidirektionale Dependency (Outer Loop → Arkwen)
Arkwen callt **keinen** Outer Loop; kein Callback/Webhook, keine Consumer-`*_url`-Config. Alle Interaktion ist consumer-initiiert (subscribe + enqueue). Fork Push-vs-Pull → **Pull**, weil der durable Stream ohnehin die Wahrheit ist und Consumer-Lag die Execution nie blockieren darf. Arkwen hat keine Compile-/Laufzeit-Abhängigkeit auf irgendeinen Outer Loop (Invariant 10).

## E2 — Zwei öffentliche Flächen, kein neuer Mechanismus
Ein Outer-Loop-Consumer ist ein headless Control-Room-Client + Queue-Producer über genau zwei versionierten Flächen:
- **Read-Plane:** subscribe/replay des Event-Streams + `get_projection(run_id, kind)`.
- **Command-Plane:** `enqueue` + `signal cancel/pause/resume` + `resolve_gate` + `reprioritize`.

Capability-bewusst (Invariant 9): `signal(cancel)` = MUSS-Contract; `signal(pause|resume)` setzt die KANN-Capability `control.pause_resume` voraus, sonst degradiert der Shim auf Workcell-Boundary-Suspension (Container-Freeze) — das paused-Overlay hält tier-unabhängig, nur die Resume-Fidelity unterscheidet sich.

## E3 — Gate-Resolution & Governance-Split
`resolve_gate` ist callback-frei (Consumer sieht `gate.requested` per subscribe, schickt die Resolution per Command). Ein **externer Resolver** ist eine per-Gate vorkonfigurierte KANN-Rolle, nur für den `escalate`-Pfad. Jedes Gate mit externem Resolver MUSS `max_wait` tragen + erbt `fail_closed` (Timeout → reject) — nie unbegrenztes Hängen an einem inter-session-Consumer. **Governance-Split:** der **Governance-Plane-Inhaber** entscheidet (Org-Floor, Portfolio-Priorität, Blueprint-Version-Constraints, Budget/Tiefe, Worker-Image-Wahl), Arkwen materialisiert & enforced. **Arkwen-intrinsischer Floor** (Redaction, fail_closed-Default, Seed-Capture) ist auch für den Governance-Inhaber NICHT abschaltbar — eine verletzende RunSpec wird beim Enqueue REJECTED, nicht still korrigiert.

## E4 — Governance-Plane-Quelle & Feedback
Der Org-Floor-Slot gehört dem **Governance-Plane-Inhaber** (ein Outer Loop wenn vorhanden, sonst Control Room/Operator — kein Consumer ist exklusiv; hält Standalone-Betrieb + Invariant 7 tier-unabhängig). Feedback = eine **Run-Metrics-Projektion** (4. Projektion, Amendment ADR-003, reiner Fold) für *jeden* Outer-Loop-/Dashboard-Consumer: `cost, duration (inkl. suspended_ms_by_reason), gate_outcomes, terminal, artifact_signals`. Best-effort & tier-degradierend — fehlende Signale absent, nie fabriziert. Scharfer Call: **Arkwen misst OBJEKTIVE Signale, der Consumer bewertet.**

## E5 — Version-Constraint & Multi-Runtime
Ein `version_constraint` (Range) des Consumers → gegen den Warehouse aufgelöst, konkrete Version/`toolkit_versions`/`image_digest` als Seeds fixiert **bei `run.created`** (Invariant 6, ADR-009 R1; nie Re-Resolve eines Range). Multi-Runtime-Payoff = „OCI für Agenten": der Outer Loop orchestriert ein Portfolio austauschbarer Worker-Typen über die uniforme Naht.

## E6 — Delivery-Semantik
`enqueue` idempotent (`idempotency_key`); subscribe at-least-once + seq-idempotent + replay; durabel + Fan-out; keine Consumer-Backpressure auf die Execution. Budget-Erschöpfung → `paused_by_resource_limit` + Consumer-Topup via `resume` (kein hartes Sterben, kein Callback; Topup hebt NUR das Kosten-Budget des Consumers, nie den physischen Org-Cap — ADR-009 R5).

## E7 — AuthZ-Minimalfloor
Read-/Command-Plane erfordern authentifizierte/autorisierte Principals; jede Command = control-Event mit `source/emitter` (`enqueue→run.created, cancel→run.cancel_requested, pause/resume→run.paused/resumed, resolve_gate→gate.resolved, reprioritize→run.reprioritized` NEU). Queue-Ordering bleibt Projektion (Invariant 2), voll auditierbar. **Volles Multi-Tenant-AuthZ** der Contract-Plane → eigenes benanntes Arbeitspaket (ADR-009 R6 — NICHT in Workcell-Secrets gefaltet).

## Transport
gRPC-first (server-stream Read + unary Command), HTTP/SSE-Fallback; Wire = ADR-002-Envelope 1:1, Forward-Compat via `schema_version`. **Spiegelbild:** aus Sicht eines Outer Loop ist ein Arkwen-Run ein **opaker Node**, gefahren von einem **consumer-seitigen `arkwen-adapter`** — der Adapter-Bau ist Sache des Consumers; Arkwen kennt keinen Consumer-Typ.

## Amendments zu früheren ADRs
- **ADR-003** 4. Projektion (Run-Metrics), reiner Fold.
- **ADR-002** Event-Stream = erstklassige externe Contract-Fläche mit Delivery-Semantik; Command-Ops = control-Events (+`run.reprioritized`).
- **ADR-005** Org-Floor benannte Quelle = Governance-Plane-Inhaber (kein Consumer exklusiv) + intrinsischer Floor darunter; externer Resolver `max_wait`+`fail_closed`.
- **ADR-001** `signal(cancel)`=MUSS, pause/resume=KANN; `version_constraint`-Auflösung bei run.created; `session_seed` vom Consumer vorgebbar.

## Invarianten-Check
Alle 10 gehalten (0 Loosening im Verify). Kern: unidirektionale Dependency (kein Consumer-Name in der Runtime), kein mutable Shadow-State (Command = Event), intrinsischer Floor per Enqueue-Reject erzwungen, keine Governance-Absorption (Arkwen misst, Consumer bewertet).

## Offene Punkte (Betrieb)
Extern-Resolver-Timeout-Default; artifact_signals-Normalisierung über Worker-Typen; Idempotency-Key-Konvention; Multi-Outer-Loop-Fairness/Fencing; Projektions-Snapshots für Consumer-Kaltstart; wie der Governance-Inhaber Warehouse-Promotion-Policy setzt.
