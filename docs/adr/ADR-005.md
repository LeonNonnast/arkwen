# ADR-005 — Quality Gate Policy Model

**Status:** akzeptiert · **Phase:** 2 · **baut auf** ADR-001 (Gate-Primitiv), ADR-003 (Suspension-Overlay)

## Kontext
Das Gate-*Primitiv* steht (ADR-001: `control.pause_resume` + `gate.interactive`; ADR-003: `gate.requested → [run.paused] → gate.resolved → [run.resumed]`, Suspension-Overlays). Dieser ADR fixiert die *Policy*-Ebene darüber: wer entscheidet, in welcher Zeit, was bei Nicht-Entscheidung passiert, und woher Gates kommen.

## Spine — ein Mechanismus, austauschbarer Resolver
*Auto und human strikt getrennt modellieren, aber über denselben Gate-Mechanismus laufen lassen.* Ein Gate wird immer gleich geboren (`gate.requested`) und beendet (`gate.resolved`); nur der **Resolver** unterscheidet sich:
- **Evaluator-Resolver (auto):** Check-Funktion / zweiter Worker / LLM-Judge; oft *am Rand* entscheidbar.
- **Human-Resolver:** reist über den Controller ins UI (Factory Floor), wartet auf einen Menschen.

**Fluss:** Shim *erkennt & raised* → Controller *evaluiert Policy & routet* → Resolver *entscheidet* → Controller schickt Entscheidung runter → Shim *injiziert* in den Worker. Controller bleibt runtime-agnostisch.

## Gate-Objekt
```
Gate = { gate_id, trigger, scope, source, mandatory,
         resolver / resolution_mode, timeout_policy, applies_to }
```

**Trigger → Suspension-Reason (ADR-003):**
| Trigger | Reason |
| --- | --- |
| Check / Tool-Permission (auto) | `paused_by_gate` |
| Menschliche Milestone-Freigabe | `paused_by_user` |
| Org-Regel (Compliance) | `paused_by_policy` |
| Budget / Ressourcen | `paused_by_resource_limit` |

## Entscheidung 1 — Granularität: `scope`
- `scope: action` — einzelner Tool-Call / Permission-Prompt (PreToolUse-artig).
- `scope: run` — Milestone: vor Deploy, Merge, Export, finalem Write-back.

## Entscheidung 2 — Timeout: **Default fail-closed**
```
timeout_policy: fail_closed | fail_open   (default = fail_closed)
max_wait: nur relevant, wenn Timeout gesetzt
```
Unaufgelöstes Gate läuft **nie** stillschweigend durch. `fail_open` nur explizit pro Gate opt-in.

## Entscheidung 3 — Resolutions-Vokabular: **reich**
`approve · reject · approve_with_modification · escalate`
- `approve_with_modification` = Mensch editiert vorgeschlagenen Diff / injiziert Guidance.
- Metadaten auf `gate.resolved`: `resolved_by · decision · rationale · resolution_payload` (Audit-Trail, Outer-Loop-Naht).

## Entscheidung 4 — Herkunft & Vorrang der Gates
Drei Quellen, materialisiert in dieser Präzedenz:
1. **Org-Policy** (nicht entfernbarer Floor)
2. **Blueprint** (ergänzt)
3. **Mission** (ergänzt)

**Regel:** Höhere Ebenen dürfen **strenger** machen, nie lockerer.

## Gate-Materialisierung pro Run — Regel vs. Instanz
**Vor Start** materialisiert Arkwen das *effektive Gate-Policy-Set* = **Regeln**, dedupliziert nach `gate_id`, mit Org-Floor. **Zur Laufzeit** wird ein `action`-Gate als **Instanz** erzeugt, sobald ein Event auf `applies_to` matcht. → versöhnt „vor Start materialisieren" mit „Action-Gates sind dynamisch".

## Konsequenzen
- `policy_version` (ADR-001-Seed) referenziert das materialisierte Set → reproduzierbar/auditierbar.
- Audit-Felder auf `gate.resolved` sind die natürliche Naht zur **Outer-Loop-/Governance-Schicht** (Phase 5).
- *(Amendment via ADR-008):* Die „Org-Policy"-Quelle wird konkret vom **Governance-Plane-Inhaber** gestellt (ein Outer Loop wenn vorhanden, sonst Control Room/Operator — kein Consumer exklusiv). Darunter liegt ein Arkwen-intrinsischer Floor (Redaction, fail_closed, Seed-Capture), der auch vom Governance-Inhaber nicht lockerbar ist (Enqueue-Reject). Ein externer Resolver (escalate) MUSS `max_wait` tragen und erbt `fail_closed`.
