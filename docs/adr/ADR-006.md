# ADR-006 — Workcell Isolation & Secrets: Volles Isolations- und Secret-Modell

**Status:** akzeptiert · **Phase:** 3 · **baut auf** ADR-001/002/003/004/005 · **reconciled durch** ADR-009

## Kontext
Threat-Model autonomer Software-Produktion: ein Worker führt selbst-generierten, nicht vorab review-baren Code aus, kann durch feindliche Materials/Tool-Ausgaben gekapert werden (Prompt-Injection-Klasse), hat offenen Handlungsraum. Drei Risikoachsen: (1) Host-/Kernel-Ausbruch, (2) Exfiltration, (3) Credential-Blast-Radius. Erweitert den Minimal-Secrets-Pfad (ADR-001) + die Workcell-Grenze (ADR-004) zum vollen Modell. Leitprinzip: **ein Mechanismus, wählbare Härtegrade; der Security-Floor degradiert — anders als beobachtbare Capabilities — NIE stillschweigend.** Controller bleibt runtime-agnostisch.

## E1 — Isolation-Profile (tiered, per Policy: `isolation_profile`)
- `standard` (**Default**): OCI-Runtime `runc`, gehärtet (eigene Namespaces, no-new-privileges, gedroppte Caps, seccomp, read-only Rootfs, non-root).
- `hardened`: gVisor (`runsc`), Syscall-Interception via User-Space-Kernel.
- `strict`: Firecracker microVM, eigener Guest-Kernel, HW-Virt-Grenze (untrusted/cross-tenant/hochsensibel).
- Linear geordnet `standard < hardened < strict`; Profilwechsel = OCI-Runtime-Klassen-Swap unter dem Cell-Shim, Arkwen-Protokoll unverändert. **Kein Auto-Downgrade:** inkompatibler Worker → fail-closed bei `provisioning` (`reason=isolation_unsatisfiable`, ein `reason` auf `failed`, kein neuer Zustand). Untrusted/multi-tenant → Org-Floor ≥ `hardened`, cross-tenant `strict`.

## E2 — Network Egress (default-deny für ALLE Profile)
Effektiv = `(Toolkit-Bedarf ∪ Blueprint-Bedarf ∪ Mission-Bedarf) ∩ Org-Ceiling [∩ Blueprint-Ceiling] − Denies`. Bedarf additiv, Governance-Ceilings NUR einschränkend — kein niedrigerer Bedarf/Ceiling überschreibt ein höheres (Invariant 7/10). Enforcement host-seitig **außerhalb der Workload-Trust-Grenze** (netns bzw. microVM vsock/tap), damit ein ausgebrochener Worker sie nicht abschalten kann. Raw-IP verboten, DNS kontrolliert. `security.egress_denied` (Host/SNI redigiert + längenbegrenzt).

## E3 — Resource-Limits (cgroups v2: cpu/mem/disk/pids/wall-clock)
Soft-Threshold → `resource.sample` + Budget-Gate (`paused_by_resource_limit`; via `approve_with_modification` NUR bis Org-Hard-Ceiling anhebbar, nie über den Org-Cap). Hard-Ceiling → kernel/shim-erzwungen → Terminal `failed reason=resource_exhausted`. Wall-Clock zählt nur running-Zeit (Suspension ausgenommen — ein auf Human-Gate wartender Run wird nicht getötet).

## E4 — Filesystem
Ephemerer Overlay (read-only Base + verworfener writable upper). overlay-upper ist INPUT für `worker.artifact_written`-Events; Workbench-Diff bleibt Fold über Events (Invariant 2), kein FS-Read. `mount` (writable Live-Bind) nur unter `standard`+trusted; unter `hardened`/`strict` fail-closed auf Snapshot gezwungen (read-only nur via virtio-fs).

## E5 — Volles Secret-Modell
Secret-Broker (control-plane, GETRENNT vom Factory Controller — Controller nie im Secret-Pfad) mintet per-Run kurzlebige, gescopte Capability-Handles/dynamische Creds; Shim fordert direkt mit per-Run-Token an. Injection bei start (env|tmpfs-file), Rotation mid-run, Leases überleben Suspension, Revocation bei reap für JEDEN Terminal-State inkl. Crash; nie im Worker Image. Broker registriert injiziertes Material AUTOMATISCH in die Cell-Shim-Redaction-Liste → ADR-004-Redaction bleibt unveränderte Persistenz-Wache. Secret-Events nur Metadaten (Invariant 4), nie Werte.

## E6 — Worker-Image-Provenance
Signatur (Sigstore/cosign) + SLSA-Provenance bei `provisioning` verifiziert, in Seed/Events (nur Refs). Hard-Enforcement Org-opt-in in Phase 3, Default in Phase 4 (Warehouse-Trust). Orthogonal zu `isolation_profile`. Phase-3-Image-Quelle = Basis-Registry (ADR-009 R9).

## E7 — Isolation-Policy-Vorrang (spiegelt ADR-005 E4)
Org-Floor > Blueprint > Mission, nur strenger. Effektiver Isolation-Contract wird vor Start dimensionsweise komponiert (profile=max, egress=∩, resource=min, image_trust=OR) und als `isolation_contract_ref` materialisiert (Regel-Projektion, kein mutable State; final in ADR-009 R2 — eigener `isolation_contract_ref`, NICHT über `policy_version`).

## Amendments zu früheren ADRs
- **ADR-001** Seed +`isolation_contract_ref, egress_policy_hash, resource_limits_hash, image_signature_ref, image_provenance_ref, secret_scope_set_ref`; Transport Socket→vsock (strict).
- **ADR-002** security-Events +`secret_leased/rotated/revoked, egress_denied`; neue `failed`-reasons `isolation_unsatisfiable, resource_exhausted`.
- **ADR-004** E3 statische Env-Injection → Secret-Broker; `mount` nur `standard`+trusted.

## Invarianten-Check
Alle 10 gehalten. Kern: Isolation ist KEINE degradierbare Capability (nur Beobachtbarkeit degradiert, Invariant 9); keine neuen Terminalzustände (Invariant 8); Secrets nie im Stream (Invariant 5, gestärkt); Controller nie im Secret-Pfad (Invariant 1).
